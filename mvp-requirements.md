# 程序员求职作战台 MVP 需求

## 1. 产品定位

面向正在找工作的程序员，提供一个本地优先的求职管理工具，用来记录投递、管理 JD、关联简历版本、沉淀面试复盘，并通过 AI 辅助分析岗位匹配点和准备方向。

一句话定位：

> 给程序员用的求职 CRM，把「投递记录 + JD 分析 + 简历匹配 + 面试复盘」放到一个工作台里。

## 2. 目标用户

第一阶段只服务一个人群：

- 中国互联网技术岗求职者
- 重点是 Android / Flutter / 前端 / 后端 / AI 应用开发等程序员
- 正在通过 BOSS 直聘、脉脉、猎头、内推、拉勾、招聘网站多渠道投递
- 希望用数据复盘投递效率，而不是凭感觉焦虑刷岗位

## 3. MVP 核心问题

程序员求职过程中常见的问题：

- 投了很多岗位，但不知道哪些渠道有效
- JD 散落在聊天记录、网页、截图里，过几天就找不到
- 不同公司聊到哪一步全靠脑子记
- 简历版本混乱，不知道哪个版本带来回复
- 面试问了什么没有系统沉淀
- 不知道自己和目标岗位差在哪里
- 每周只感到焦虑，但没有数据复盘

MVP 要解决的不是「自动帮你找到工作」，而是：

> 让求职过程可记录、可分析、可复盘、可优化。

## 4. 第一版功能范围

### 4.1 岗位管理

用户可以新增一个岗位记录，字段包括：

- 公司名称
- 岗位名称
- 投递渠道
- 城市 / 远程
- 薪资范围
- JD 原文
- 岗位链接
- 当前状态
- 投递日期
- 下次跟进日期
- 备注

状态建议：

- 待评估
- 已投递
- 已沟通
- 已约面
- 一面
- 二面
- HR 面
- Offer
- 已拒绝
- 无反馈
- 不合适

### 4.2 看板视图

用 Kanban 看板展示岗位状态。

列：

- 待评估
- 已投递
- 沟通中
- 面试中
- Offer
- 结束

支持拖拽改状态可以后置，MVP 第一版可以先用下拉选择状态。

### 4.3 JD 技术关键词分析

用户粘贴 JD 后，系统提取：

- 技术栈关键词
- 业务领域关键词
- 岗位能力关键词
- 加分项
- 风险提示

示例输出：

```text
技术栈：Flutter、Dart、Android、Kotlin、性能优化
业务领域：出海、金融、交易
能力要求：工程化、稳定性、跨端、多语言
加分项：Google Play 上架、CI/CD、AI Coding
风险提示：要求 5 年以上团队管理经验，当前简历需突出项目主导经历
```

MVP 可先用规则词库 + AI 调用。没有 AI 时，也能基于关键词给出粗分析。

### 4.4 简历版本管理

用户可以保存多个简历版本：

- 简历名称
- 适用方向
- 简历正文或文件路径
- 核心卖点
- 创建时间

岗位可以关联一个简历版本，用于后续复盘：

> 哪个简历版本带来了更多回复和面试。

### 4.5 JD 与简历匹配建议

输入：

- 当前岗位 JD
- 关联简历版本

输出：

- 匹配点
- 缺失点
- 建议突出的项目
- 建议补充的关键词
- BOSS / 脉脉打招呼话术
- 面试准备方向

MVP 不追求 ATS 分数，重点做「可行动建议」。

### 4.6 面试记录与复盘

每个岗位可以添加多条面试记录：

- 面试轮次
- 面试时间
- 面试官类型
- 问题记录
- 自我评价
- 需要补强的知识点
- 结果
- 复盘总结

面试问题可以打标签：

- 项目经历
- 基础知识
- 架构设计
- 性能优化
- Flutter
- Android
- 算法
- 系统设计
- HR

### 4.7 数据统计

MVP 第一版统计：

- 总投递数
- 本周投递数
- 回复数
- 面试数
- Offer 数
- 投递到回复转化率
- 回复到面试转化率
- 渠道效果统计
- 状态分布

后续可增加：

- JD 技术关键词出现频率
- 哪类岗位回复率最高
- 哪个简历版本效果最好

## 5. 非目标

MVP 不做：

- 自动海投
- 爬取招聘网站
- 自动填写申请表
- 复杂 ATS 分数
- 多用户协作
- 企业招聘端
- 社交网络
- 在线支付

这些会让项目变重，也容易和成熟产品正面竞争。

## 6. 推荐技术方案

第一版建议做本地优先工具：

- 前端：React + Vite + TypeScript
- UI：普通 CSS 或轻量组件库
- 数据：SQLite
- 桌面壳：Tauri 可选
- AI：先预留 OpenAI / Ollama Provider 接口

如果想最快验证，也可以先做纯 Web + localStorage / IndexedDB。

推荐第一版：

```text
React + TypeScript + IndexedDB
```

原因：

- 开发快
- 不需要后端
- 方便部署到 GitHub Pages
- 也方便后续包成 Tauri 桌面端

## 7. MVP 成功标准

不是看 star，而是看它是否真的帮助求职。

第一版成功标准：

- 可以记录至少 50 个岗位
- 可以清楚看到每个岗位状态
- 可以保存 JD 和面试记录
- 可以统计每周投递和回复情况
- 可以基于 JD 生成一份有用的匹配建议
- 用户愿意连续使用 2 周
