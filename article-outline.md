# 配套文章大纲

## 标题

AI Demo 到真正上线，中间差的是移动端工程化：我做了一个程序员求职作战台

## 目标读者

- 正在使用 Cursor / Codex / Claude Code 写项目的移动端开发者
- Android / Flutter 工程师
- 正在求职、转型 AI 应用开发、需要作品背书的程序员

## 核心观点

AI 已经能很快生成 App Demo，但真正可上线、可维护、可交付的产品，还需要工程化闭环。工程师的价值不只在写页面，而在把需求拆成可验证切片，把数据、状态、边界、交互、备份和演示链路串起来。

## 文章结构

### 1. AI Demo 为什么看起来很快

- 自然语言可以快速生成页面、表单、列表和样式
- 小范围功能验证速度很高
- 适合做原型、验证想法、补齐样板代码

### 2. 真正上线还差哪些工程化步骤

- 需求切片：先做能闭环的最小版本
- 数据模型：岗位、简历、面试记录之间要有关系
- 本地持久化：刷新页面后数据不能丢
- 状态管理：新增、编辑、删除、筛选、选中态都要稳定
- 边界处理：空状态、导入格式错误、无 JD、无简历等
- 可演示性：示例数据、README、截图、文章说明

### 3. 为什么我选择做求职 CRM

- 求职者真实存在投递记录分散、JD 难复盘、简历版本混乱的问题
- 它不依赖复杂后端，适合纯 Web 本地 MVP
- 它能展示移动端工程师的产品意识、工程拆解和 AI 协作能力

### 4. MVP 功能拆解

- 岗位管理：记录公司、岗位、渠道、状态、跟进日期
- JD 分析：提取技术关键词、风险点和岗位画像
- 简历版本：维护不同方向的简历内容和亮点
- 简历匹配：根据 JD 和简历生成匹配点、缺失点、沟通话术
- 面试复盘：记录轮次、问题、标签、薄弱点和总结
- AI Prompt Pack：不配置 API Key，也能生成面试准备和参考答案 Prompt
- 导入导出：用 JSON 做本地备份和演示迁移

### 5. 技术实现

- React + TypeScript + Vite
- IndexedDB 本地存储
- 规则引擎优先，后续再接 AI Provider
- 零配置 Prompt Pack 优先，避免早期产品承担模型调用成本
- 可选 AI Provider 后置，支持用户自带 API Key 或 Ollama
- 功能按 Slice 迭代，先保证每个切片可运行、可验证

### 6. 这个项目能证明什么

- 能把模糊需求拆成可交付功能
- 能设计本地数据结构和前端状态流
- 能围绕真实场景做产品化取舍
- 能用 AI 提效，但保留工程师的判断和验收能力

## 结尾

AI 会降低写代码的门槛，但不会自动补齐工程化交付。未来移动端程序员的竞争力，可能不只是会写 Android / Flutter，而是能把 AI 生成的半成品推进到可上线、可维护、可复盘的产品形态。
