# 我做了一个给程序员用的本地求职 CRM

> 粘贴 JD、关联简历版本、记录面试复盘，再用数据看看自己到底卡在哪里。

最近我发现，程序员找工作时最累的地方，不只是写简历和刷题，而是信息太分散。

一个岗位在 BOSS 上聊过，JD 复制到微信里；另一个岗位来自内推，简历发的是某个临时改出来的版本；面试问了哪些问题，当天还能记得，过几天就只剩一个模糊印象。投递多了以后，焦虑感会上升，但数据感很弱：到底哪个渠道有效？哪个简历版本更容易拿面试？自己反复薄弱的是算法、项目表达，还是系统设计？

所以我做了一个小工具：**程序员求职作战台**。

它不是给 HR 用的 ATS，也不是企业 CRM。它只服务一个人：正在求职、同时管理很多岗位的开发者。

## 它解决什么

我想把求职过程里的核心循环串起来：

```text
记录岗位 -> 分析 JD -> 关联简历 -> 跟踪面试 -> 复盘数据
```

目前已经可以做这些事：

- 粘贴 JD 后自动提取公司、岗位、渠道、城市、薪资和工作方式
- 管理岗位状态，比如已投递、面试中、Offer、已拒绝、无反馈
- 保存 JD 原文，避免岗位下架后找不到信息
- 管理多个简历版本，并把投递岗位关联到具体简历
- 用本地规则分析 JD 关键词，生成简历匹配建议
- 记录面试邀约、问题、复盘、薄弱点和行动项
- 在统计页看投递漏斗、渠道分布、瓶颈提示和高频薄弱点
- 导入导出 JSON，数据默认留在浏览器本地

## 为什么强调本地优先

求职数据其实很私密。

JD、简历、薪资、面试反馈、失败原因，这些东西不太适合默认上传到某个服务端。所以这个项目一开始就做成本地优先：不登录、不需要后端、不做云同步，数据保存在当前浏览器 IndexedDB。

AI 也是可选的。没配置 API Key 时，工具仍然可以用本地规则和 Prompt Pack；如果你配置 OpenAI-compatible API 或 Ollama，它才会调用外部模型。

## 我觉得最有价值的不是“记录”，而是复盘

很多求职工具会停在“把岗位记下来”。但真正影响效率的是后面的复盘：

- 哪些渠道回复率更高？
- 哪类岗位更容易进入面试？
- 哪些面试问题反复出现？
- 哪个简历版本更适合当前 JD？
- 自己的薄弱点是技术不会，还是表达没组织好？

如果这些问题能被持续记录下来，求职就不只是“继续投”，而会变成一个可迭代的系统。

## 适合谁

这个项目更适合这些人：

- Android / Flutter / 前端 / 后端 / AI 应用开发者
- 正在通过 BOSS、脉脉、内推、猎聘、拉勾等渠道求职的人
- 同时投递多个岗位，需要管理 JD、简历版本和面试反馈的人
- 想用数据复盘求职效率，而不是只凭感觉焦虑的人

不适合企业招聘团队，也不做批量投递、招聘网站爬虫、多人协作和云同步。

## 可直接发社区的短版

我开源了一个「程序员求职作战台」。

它是一个本地优先的求职 CRM：粘贴 JD 自动建档，记录投递状态，关联简历版本，沉淀面试问题和复盘，再用统计页看渠道转化、瓶颈和高频薄弱点。

不是给 HR 用的 ATS，也不做批量投递和爬虫。目标很简单：帮正在求职的开发者把散落在招聘 App、微信、简历文档和 AI 对话里的信息收回来。

默认无登录、无后端、无云同步，数据存在浏览器 IndexedDB。AI 是可选的，没 API Key 也能用本地规则和 Prompt Pack。
